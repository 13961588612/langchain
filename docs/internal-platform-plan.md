> **定位**：公司内部使用的智能体管理开发平台
> **核心用户**：AI 开发团队（创建/管理/编排智能体） + 业务团队（使用智能体对话）

## 1. 定位与目标用户

```
公司内部平台 = 智能体"工场"

  ┌─────────────────────────────────────────────┐
  │            AetherMind (Internal)             │
  │                                             │
  │  AI 开发团队 (Developer)                      │
  │  ├── 创建/配置智能体                          │
  │  ├── 编写 agent.md / soul.md / profile.md    │
  │  ├── 开发/安装 Skills                        │
  │  ├── 管理模型配置                             │
  │  ├── 设计多智能体编排                          │
  │  ├── 调试/追踪智能体行为                       │
  │  └── 配置审批安全策略                          │
  │                                             │
  │  业务团队 (User)                              │
  │  ├── 通过 Web UI 与智能体对话                  │
  │  ├── 通过企业微信/飞书/钉钉 使用智能体          │
  │  └── 查看对话历史                             │
  │                                             │
  │  管理员 (Admin) — 简化版                      │
  │  ├── 用户管理（内部用户 CRUD）                 │
  │  ├── 整体用量监控                             │
  │  └── 系统健康检查                             │
  └─────────────────────────────────────────────┘

暂不涉及（内部平台不需要）：
  ✗ 多租户隔离（单公司，不需要）
  ✗ 复杂计费系统（内部结算）
  ✗ Skills 公开市场（内部 Skills 库即可）
  ✗ OAuth2 自建体系（简化认证）
  ✗ CI/CD 流水线（初期手动部署）
  ✗ Vault 密钥管理（环境变量 + .env 管理）
  ✗ Feature Flags（不需要灰度）
  ✗ GDPR 合规（内部工具）
  ✗ 多节点高可用（单机/小规模）
```

### Phase 2：智能体灵魂系统（Soul & Identity）

**目标**：让每个智能体拥有独立人格和身份定义，而不仅仅是 system_prompt 文本。

#### 2.1 Soul 系统

```
Soul = 人格核心，持久定义智能体"是谁"

soul.md 格式:
  ---
  name: "客服助手小A"
  role: "customer_support"
  persona: "友好、耐心、专业"
  tone: "warm_professional"        # warm/cold/creative/analytical/direct
  style: "简洁明了，避免术语"
  values:
    - "用户满意度第一"
    - "诚实透明"
    - "主动帮助"
  boundaries:
    - "不提供医疗建议"
    - "不讨论政治话题"
  quirks:
    - "喜欢用生活化的比喻"
    - "偶尔用表情符号调节气氛"
  ---

  ## 自我介绍
  我是小A，你的专属客服助手。无论遇到什么问题...

Soul 模板库（内置）:
  • 助手型 (assistant)     — 通用帮助
  • 导师型 (tutor)         — 教学引导，苏格拉底式
  • 分析师型 (analyst)     — 数据驱动，严谨推理
  • 创意型 (creative)      — 发散思维，脑暴
  • 客服型 (support)       — 友好耐心，解决问题
  • 技术顾问型 (tech_lead) — 代码 review，架构建议
```

#### 2.2 Profile / Behavior 配置

```
profile.md:
  控制智能体的"行为偏好"

  response_length: "concise"       # concise/balanced/detailed
  tool_usage: "proactive"          # proactive / on_demand / cautious
  initiative: "medium"             # 主动程度 low/medium/high
  follow_up: true                  # 对话结束时是否追问
  language: "zh-CN"                # 默认语言
  greeting_style: "warm"           # 开场方式

Profile 通过 DeepAgents middleware 或 system_prompt 注入生效
```

Agent 构建流程:

        fallback_2: qwen2.5:72b
        final: gpt-4o-mini

  Agent 创建时选择路由策略 → ModelHub 执行路由

#### 4.3 Memory 系统

```
Memory 分层:

  Layer 1: 对话上下文 (Checkpointer)
    → LangGraph PostgresSaver（已实现）
    → 当前对话窗口: 最近 20 条消息

  Layer 2: 对话摘要
    → 每 10 轮或 > 4000 tokens 触发 LLM 摘要
    → 摘要注入下一轮对话

  Layer 3: 长期记忆 (Vector)
    → 关键信息向量化存储（pgvector）
    → 按语义相关性检索注入当前对话

### Phase 5：Skills 开发工场

**目标**：让 AI 开发团队能够开发、测试、管理内部 Skills，这是"开发平台"的核心价值。

#### 5.1 Skills 定位（内部版）

```
内部 Skills ≠ 公开市场。核心场景：

  1. 开发者编写 Skill → 安装到自己的 Agent 测试 → 发布到内部库
  2. 团队成员浏览内部库 → 一键安装到自己的 Agent
  3. 每个 Agent 可安装多个 Skills，按优先级加载

不涉及（内部平台不做）：
  ✗ 公开市场发布
  ✗ 用户评分/评论系统
  ✗ 社区审核流程
```

#### 5.2 Skill 开发流程

```
开发流程:

  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
  │ 创建     │ → │ 编写     │ → │ 测试     │ → │ 发布     │
  │ Skill   │    │ SKILL.md │    │ 到 Agent │    │ 到内部库 │
  └─────────┘    └─────────┘    └─────────┘    └─────────┘

创建 Skill 的方式:

  方式 1: 前端 Skill 编辑器
    → 填表单（名称/描述/触发条件）+ 写 SKILL.md 内容 + 上传资源文件
    → 保存到 Agent 的 skills/ 目录

  方式 2: 直接从文件系统导入
    → 把 SKILL.md 放到 Agent 工作目录 → 自动识别

  方式 3: 从内部库安装
    → 浏览已发布的 Skills → 一键安装
```

              └── 无匹配 → 通用模式

  Skills 来源优先级:
    1. Agent Skills (agent's skills/ 目录)    ← 专用
    2. Project Skills (暂不做)                 ← 项目级
    3. User Skills (暂不做)                    ← 用户级

Skills 热加载:
  • Agent 运行时修改 skills/ 目录
  • AgentRuntime.clear_cache(agent_id) 清除缓存
  • 下次对话自动加载新 Skill
```

### Phase 6：多智能体编排

**目标**：让开发团队能设计和运行多智能体协作流程。

#### 6.1 编排模式（当前聚焦 Supervisor）

```
三阶段实现:

  第一批（当前）: Supervisor 模式
    Supervisor Agent 分析任务 → 分发给专业子 Agent → 汇总结果

  第二批: Swarm / Handoff
    智能体间任务移交，A 处理不了 → 转给 B

  第三批: Map-Reduce
    大任务拆解 → 并行执行 → 汇总

先只做 Supervisor——覆盖 80% 场景。
```

## 4. 简化认证方案（内部版）

```
内部平台 → 不需要 OAuth2 自建体系

简化方案:

  Level 1: API Key（Phase 1 收尾就做）
    → 后端生成固定 API Key（如 AETHERMIND_API_KEY）
    → 前端/API 调用在 Header 中携带
    → 仅防未授权访问，不涉及用户身份

  Level 2: 简易用户系统（Phase 2 或 3 补）
    → User 模型: email / password_hash / role / is_active
    → Login → JWT (长期有效，如 30 天)
    → 角色: admin / developer / user
    → 不需要 MFA、不需要 OAuth、不需要 SSO

  Level 3: 企业 SSO（企业加固批次）
    → LDAP / OIDC 集成
    → 与企业统一认证打通

当前先做 Level 1，Phase 2-3 补 Level 2。Level 3 在企业加固批次做。
```
