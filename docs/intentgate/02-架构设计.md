# 02 架构设计

## 1. 设计原则

1. **卡片是交互协议**：Agent 产出 `CardIntent`（模板名 + 槽位），Gateway 负责渲染与投递。
2. **时限事件不进 Agent**：按钮、分页、简单确认走 Gateway 快路径。
3. **Gateway 独立进程**：WebSocket 心跳、5 秒 update、限流与 LLM 解耦。
4. **通道与 Agent 双适配**：Channel Adapter 翻译 IM 协议；Agent Adapter 翻译智能体协议。
5. **状态在 Gateway**：`task_id`、会话上下文、活跃卡片由 Session Store 管理，Agent 无状态可水平扩展。

## 4. 数据流

### 4.1 用户发文本（慢路径）

```
用户 @ 机器人发消息
    → WeComAdapter 收到 aibot_msg_callback
    → 构造 StandardMessage { session_id, channel, user_id, text, metadata }
    → Gateway 可选：先发 stream「正在处理…」
    → AgentBackend.chat(message)  → SSE 流 AgentReply
    → Gateway 按 kind 分发：
         text      → aibot_respond_msg (text)
         stream    → aibot_respond_msg (stream, 同一 stream.id)
         card      → CardRuntime.render → aibot_respond_msg (template_card)
         markdown  → aibot_respond_msg (markdown)
    → 若 kind=card：SessionStore.save_card(task_id, intent)
```

### 4.2 用户点卡片按钮（快路径优先）

```
用户点击按钮
    → WeComAdapter 收到 template_card_event
    → 构造 CardActionEvent { task_id, action_key, ... }
    → EventRouter.is_fast_path(action_key)?
         │
    是 ──┴──► SessionStore.get_card(task_id)
              → 执行业务规则（如 approve → 调 CRM API）
              → CardRuntime 渲染新状态
              → aibot_respond_update_msg（<5s）
              → SessionStore.update_card_state
         │
    否 ──┴──► 5s 内 update 占位卡「处理中…」
              → AgentQueue 异步 AgentBackend.handle_action(event)
              → 完成后 aibot_send_msg 新卡 或 update
```

### 4.3 主动推送

```
业务 / Agent 异步任务完成
    → Gateway Outbound.send_proactive(session_id, AgentReply)
    → 校验：用户须曾在该会话发过消息（企微限制）
    → CardRuntime.render → aibot_send_msg
```

## 5. 快路径 vs 慢路径

```
                    CardActionEvent
                          │
                          ▼
              action_key ∈ FAST_PATH_ACTIONS?
                    /              \
                  是                否
                  │                  │
                  ▼                  ▼
         规则引擎 + CRM API    占位 update（时限内）
         直接 update 卡片            │
                  │                  ▼
                  │           AgentBackend.handle_action
                  │                  │
                  └────────┬─────────┘
                           ▼
                    更新卡片 / 主动推新卡
```

**快路径白名单**（当前代码 `gateway/router.py`）：

`approve`、`reject`、`confirm`、`cancel`、`page_next`、`page_prev`

扩展方式：配置文件或数据库维护 `{ action_key → handler }` 映射，无需改 Agent。

| 路径 | 典型场景 | 调用 LLM |
|------|----------|----------|
| 快路径 | 审批、取消、分页 | 否 |
| 慢路径 | 改参数、复杂改单、需推理 | 是 |

## 6. 会话与标识

### 6.1 session_id

```
session_id = {channel}:{tenant}:{conversation_key}
```

示例：

```
wecom:default:bot123_gchat456_2     # 群聊 chat_type=2
wecom:default:bot123_user789_1      # 单聊
web:default:conv_uuid               # Web 对话
```

`conversation_key` 由 Channel Adapter 按客户端规则拼接（如 `botId + chatid + chatType`）。

### 6.2 task_id

```
task_id = {业务}:{bizId}:{version}
```

示例：`crm:approve:12345:v1`

规则：

- 发卡片时写入 Session Store
- 点击时 O(1) 查状态，**不解析自然语言**
- update 时必须与发卡时 **完全一致**

### 6.3 Session Store 结构

```json
{
  "session_id": "wecom:default:bot1_chat1_2",
  "channel": "wecom",
  "chatid": "CHATID",
  "chat_type": 2,
  "user_id": "userid",
  "last_req_id": "req_xxx",
  "last_user_msg_at": "2026-05-20T10:00:00Z",
  "active_cards": {
    "crm:approve:12345:v1": {
      "template": "crm_approve",
      "biz_type": "crm_approve",
      "biz_id": "12345",
      "state": "pending"
    }
  }
}
```

生产环境使用 Redis Hash + TTL；开发使用 `InMemorySessionStore`。

## 7. Channel Adapter 接口（设计）

每个 Channel Adapter 实现以下能力：

```python
class ChannelAdapter(ABC):
    channel: ChannelType

    async def start(self) -> None:
        """启动连接（WS 监听 / HTTP server）"""

    async def stop(self) -> None:
        """优雅关闭"""

    def normalize_message(self, raw: dict) -> StandardMessage | None:
        """原生入站 → StandardMessage"""

    def normalize_card_event(self, raw: dict) -> CardActionEvent | None:
        """原生卡片事件 → CardActionEvent"""

    async def send_reply(self, ctx: OutboundContext, reply: AgentReply) -> None:
        """AgentReply → 原生出站"""

    async def update_card(self, ctx: OutboundContext, intent: CardIntent) -> None:
        """时限内更新已发卡片"""

    async def send_proactive(self, session_id: str, reply: AgentReply) -> None:
        """主动推送"""
```

`OutboundContext` 含 `req_id`、`chatid`、`chat_type` 等通道特有字段，从 Session Store 读取。

## 8. Agent Backend 接口

```python
class AgentBackend(ABC):
    async def chat(self, message: StandardMessage) -> AsyncIterator[AgentReply]: ...
```

## 9. 部署拓扑

### 开发

```
单进程：uvicorn intentgate.main:app --port 8090
       + WeComAdapter 同进程或子进程（P1）
       + AgentScope 本地 8100
       + Redis 可选
```

### 生产（推荐）

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ IntentGate   │     │ AgentScope   │     │ Redis        │
│ Gateway x1   │────►│ Runtime xN   │     │ Session      │
│ (单 Bot 单连接)│     │              │     └──────────────┘
└──────────────┘     └──────────────┘
```

约束：

- **每个企微 Bot 仅一个 Gateway 实例**维持长连接
- Agent 后端可水平扩展
- Gateway 与 Agent 之间消息队列解耦长任务

## 10. 反模式

| 反模式 | 后果 | 正确做法 |
|--------|------|----------|
| 每次点击跑完整 Agent | 超 5s，update 失败 | 快路径 + 异步补发 |
| LLM 输出 template_card JSON | 结构错、难维护 | CardIntent + YAML |
| stream 与 card 同包 | 企微不支持 | 先 finish stream 再发卡 |
| 无 task_id | 无法 update | 发卡时写 Store |
| Gateway 与 Agent 同进程 | 心跳断连 | 独立部署 |
| 忽略频率限制 | 静默失败 | Outbound 队列 + 合并 |

## 11. 扩展点

| 扩展点 | 方式 |
|--------|------|
| 新 IM 通道 | 实现 `ChannelAdapter` + `CardRenderer` |
| 新 Agent 后端 | 实现 `AgentBackend` |
| 新卡片场景 | 新增 `cards/*.yaml` |
| 新快路径动作 | 配置 `action_handlers` |
| 业务回调 | 快路径 handler 内调 HTTP/gRPC |
