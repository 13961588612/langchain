## Phase 2：智能体灵魂系统（Soul & Identity）

**目标**：每个 Agent 拥有独立人格与行为偏好，而非单一 `system_prompt` 文本。

### 2.1 soul.md 格式

```markdown
---
name: "客服助手小A"
role: customer_support
persona: "友好、耐心、专业"
tone: warm_professional
style: "简洁明了，避免术语"
values:
  - "用户满意度第一"
  - "诚实透明"
boundaries:
  - "不提供医疗建议"
quirks:
  - "偶尔用生活化比喻"
---

## 自我介绍
我是小A，你的专属客服助手...
```

**内置 Soul 模板**：assistant、tutor、analyst、creative、support、tech_lead

### 2.2 profile.md 格式

```yaml
response_length: concise      # concise | balanced | detailed
tool_usage: proactive         # proactive | on_demand | cautious
initiative: medium
follow_up: true
language: zh-CN
greeting_style: warm
```

### 2.3 Prompt Assembler

组装顺序：

```
[Soul Identity] → [Behavior Profile] → [Tools Available] → [Base Instructions]
```

实现：`backend/app/core/prompt_assembler.py`（待建）

### 2.4 前端

Agent 编辑页新增 **「灵魂 Soul」** Tab：模板选择、字段表单、实时预览组装后的 system_prompt。

### 2.5 任务清单

- [ ] soul.md / profile.md 解析器（YAML frontmatter）
- [ ] PromptAssembler 模块
- [ ] 6 种 Soul 模板 seed 数据
- [ ] 前端 Soul 编辑器 Tab
- [ ] 创建 Agent 时可选 Soul 模板

## Phase 3：Model Hub 模型中心

**目标**：统一管理公司 LLM 资源，Agent 选模型不关心 Provider 细节。

### 3.1 管理界面

- Provider 列表：OpenAI / Anthropic / Google / Azure / Ollama
- 每 Provider 下模型列表、连接状态、API Key 配置
- 「扫描本地模型」：Ollama / vLLM 端点
- 模型标签：推荐 / 快速 / 推理 / 编程 / 长文本

### 3.2 ModelConfig 能力

- CRUD API + 前端配置页
- API Key AES 加密存储
- `POST /api/models/{id}/test` 连接测试
- 默认参数：temperature、max_tokens、top_p

### 3.3 路由策略（Agent 级）

| 策略 | 行为 |
|------|------|
| `fixed` | 固定指定模型 |
| `cheapest_first` | 优先便宜模型 |
| `best_quality` | 默认最强模型 |
| `fallback_chain` | 主模型失败依次降级 |

示例 fallback：

```yaml
primary: gpt-4o
fallback_1: claude-sonnet
fallback_2: qwen2.5:72b
final: gpt-4o-mini
```

- [ ] 路由策略执行器
- [ ] 前端 Model Hub 页
- [ ] Agent 创建/编辑集成模型选择器

## Phase 4：工作目录系统

**目标**：可编辑、可版本化的 Agent 工作区，作为「智能体开发 IDE」核心。

### 4.1 目录结构

```
workdirs/{agent_id}/
├── agent.md           # 智能体定义
├── soul.md
├── profile.md
├── memory.md          # 长期记忆快照
├── tools.yaml         # 工具声明
├── skills/
├── data/
│   ├── knowledge/
│   └── cache/
└── .versions/
    ├── agent.md.v1
    └── manifest.json
```

### 4.2 前端 IDE

- 左文件树 + 右编辑区
- YAML frontmatter 语法高亮与校验
- tools.yaml JSON Schema 补全
- 版本历史：diff、回滚、注释

### 4.3 Memory 三层

| 层 | 机制 | 状态 |
|----|------|------|
| L1 短期 | PostgresSaver Checkpointer | ✅ 已有 |
| L2 中期 | 每 10 轮或 >4000 tokens LLM 摘要 | 待建 |
| L3 长期 | pgvector 向量检索 + memory.md 快照 | 待建 |

前端 Memory 管理：查看/编辑/删除/标记重要度。

### 4.4 tools.yaml

声明式工具配置 + 允许/禁止列表，与 DeepAgents 工具注册联动。

### 4.5 任务清单

- [ ] WorkdirManager 抽象（Local FS → 后期 MinIO）
- [ ] 文件 CRUD API
- [ ] 版本 diff/rollback

## Phase 5：Skills 开发工场

**目标**：团队开发、测试、共享内部 Skills。

### 5.1 定位

- **内部库**，非公开市场
- 流程：创建 → 编写 SKILL.md → 安装到 Agent 测试 → 发布到内部库

### 5.2 SKILL.md 规范

遵循 DeepAgents / Anthropic Agent Skill 约定：frontmatter（名称、描述、触发词）+ Markdown 正文。

### 5.3 创建方式

1. 前端 Skill 编辑器（表单 + Markdown）
2. 直接放入 `workdirs/{id}/skills/`
3. 从内部库一键安装

### 5.4 运行时

- 触发匹配 → 加载 Skill 注入上下文
- 来源优先级：Agent skills/ 目录
- 热加载：`clear_agent_cache(agent_id)`

### 5.5 内置 Skills（计划）

| Skill | 说明 |
|-------|------|
| web-search | Tavily 搜索 |
| code-analysis | 代码分析 |
| doc-qa | 文档问答 |
| data-analysis | 数据分析 |

## Phase 6：多智能体编排

**目标**：Supervisor 模式多 Agent 协作（优先覆盖 80% 场景）。

### 6.1 编排模式路线图

| 批次 | 模式 | 说明 |
|------|------|------|
| 1 | Supervisor | 分发 + 汇总 |
| 2 | Swarm / Handoff | Agent 间移交 |
| 3 | Map-Reduce | 并行子任务 |

### 6.2 YAML 配置示例

```yaml
name: "代码审查团队"
mode: supervisor

supervisor:
  agent: "tech-lead"
  instructions: "分析变更，分发给审查者"

members:
  - agent: "code-style-checker"
    role: "代码风格"
  - agent: "security-reviewer"
    role: "安全审查"

routing:
  - pattern: ".*style|lint.*"
    target: "code-style-checker"
  - pattern: ".*security|vuln.*"
    target: "security-reviewer"
  - pattern: ".*"
    target: "code-style-checker"
```

### 6.3 编排 SSE 事件

| event | 说明 |
|-------|------|
| `orchestration_start` | 开始 |
| `agent_start` | 子 Agent 启动 |
| `dispatch` | Supervisor 分发 |
| `agent_result` | 子 Agent 结果 |
| `aggregate` | 汇总 |
| `orchestration_done` | 结束 |

- [ ] 前端编排列表/编辑/运行历史

## Phase 7：Human-in-the-Loop

**目标**：敏感操作人工审批，Agent 行为可控。

### 7.1 审批规则（Agent 级）

| 操作 | 默认 |
|------|------|
| write_file | 需审批 |
| execute_command | 需审批 |
| delete_file | 可选 |
| send_message | 通常不需要 |

配置：审批人、超时（如 5 分钟）、超时策略（拒绝/自动通过）、通知渠道（Web / 企微经 IntentGate）。

### 7.2 流程

```
Agent 遇敏感 tool → 暂停 → 创建 Approval 记录
    → 通知审批人 → 批准/拒绝/超时
    → 恢复或中止 Agent 执行
```

### 7.3 前端

待审批队列、上下文查看、批准/拒绝、历史记录。

### 7.4 任务清单

- [ ] Approval ORM + 状态机

- [ ] 审批 API + WebSocket/SSE 通知
- [ ] 前端审批面板
- [ ] IntentGate 推送审批卡片（可选）

## Phase 9：多通道接入（IntentGate）

**目标**：企微/飞书/钉钉使用智能体，**卡片交互**经 IntentGate。

**为何最后做**：Web UI 已满足内部开发；IM App 申请流程长；核心 Agent 能力应先完善。

### 9.1 架构选择

不在 AetherMind 重复实现 IM Gateway，而是：

```
IM → IntentGate → AetherMindBackend → /api/agents/{id}/chat
```

AetherMind 侧工作：

1. SSE 增加 `card_intent` 事件
2. DeepAgents 注册 `show_card` 工具
3. `Channel` 表配置 agent_id 供 IntentGate 路由
4. （可选）Web Channel 使用 IntentGate WebRenderer 组件

### 9.2 任务清单

- [ ] `show_card` 工具 + card_intent SSE
- [ ] `POST /api/agents/{id}/actions`（慢路径卡片）
- [ ] Channel 管理页绑定 IntentGate
- [ ] 与 IntentGate P4 联调

详见 [06-IntentGate集成](./06-IntentGate集成.md)。
