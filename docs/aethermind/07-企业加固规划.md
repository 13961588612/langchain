# 07 企业加固规划

> **状态：延后批次。** 当前优先完成内部开发平台 Phase 2–9。本文档供企业化部署前参考，不要求与内部版同步实现。

---

## 1. 触发条件

以下情况启动企业加固批次（Phase E1–E3）：

- 平台对外或多部门隔离使用
- 需通过安全审计 / 合规检查
- 单机 Docker 无法满足 SLA
- 需与 LDAP/OIDC 统一身份对接

## 3. Phase E1：企业基础设施

### 3.1 多租户（Shared DB + RLS）

- 模型：`Tenant` → `User`（多对多 `user_tenants`）
- 所有业务表含 `tenant_id`
- PostgreSQL RLS：`USING (tenant_id = current_setting('app.current_tenant_id')::uuid)`
- API 请求头 `X-Tenant-ID` 与 JWT claims 交叉校验

租户级隔离：LLM Key、模型配额、Agent 数量、存储、通道绑定。

### 3.2 认证授权（OAuth2 + JWT）

| Token | 有效期 | 用途 |
|-------|--------|------|
| Access Token | 15 min | API 鉴权 |
| Refresh Token | 7 days | 刷新 Access |
| API Key | 可撤销 | 程序化调用 |

**RBAC 角色**（租户内）：

| 角色 | 权限 |
|------|------|
| admin | 全部 |
| developer | Agent/Skill/Model 管理 |
| operator | 启停、监控 |
| user | 使用已部署 Agent |

权限域：`agents:*`、`skills:*`、`models:*`、`channels:*`、`conversations:*`、`admin:*`

API Key 格式：`ak_{tenant}_{random32}`，AES-256-GCM 加密存储。

### 3.3 API 网关（Traefik）

- TLS 终止、JWT 验证、限流、CORS、熔断
- 限流示例：全局 300 req/min/IP，租户 1000 req/min，chat 30 req/min/user

### 3.4 密钥管理（Vault）

- kv-v2 静态密钥、transit 加密、database 动态凭证
- AppRole 服务间认证

### 3.5 审计日志

事件模型：`event_id`、`tenant_id`、`actor`、`action`、`resource`、`result`、`changes`  
存储：ES 热数据 90 天 → PG 温数据 → 冷归档。

## 4. Phase E2：质量与交付

### 4.1 数据库迁移

Alembic 替代 `Base.metadata.create_all()`，CI 阶段 `alembic check`。

### 4.2 测试金字塔

| 层级 | 比例 | 工具 |
|------|------|------|
| 单元 | 60% | pytest + pytest-asyncio |
| 集成 | 30% | httpx + testcontainers |
| E2E | 10% | Playwright |

核心模块覆盖率目标 >90%，整体 >75%。

### 4.3 CI/CD（GitLab CI 示例）

1. Lint & Type Check（ruff、mypy、eslint）
2. Unit Tests
3. Integration Tests（含租户隔离安全测试）
4. Security Scan（Trivy、Bandit、trufflehog）
5. Build & Push 镜像
6. Staging 部署 + 冒烟
7. Production 手动触发 + Canary 10%→50%→100%

### 4.4 错误响应标准化

统一 `APIErrorResponse`：`code`、`message`、`request_id`、`timestamp`。

### 4.5 健康检查增强

`/api/health/live`、`/ready`、`/startup`，含 database、redis、vault、LLM provider 分项状态。

## 5. Phase E3：高可用与灾备

### 5.1 部署拓扑

- Traefik 多节点 + 无状态 FastAPI 水平扩展
- PostgreSQL Patroni + etcd 主备
- Redis Sentinel
- MinIO 分布式或 NFS 共享 workdirs

### 5.2 备份

| 对象 | 策略 |
|------|------|
| PostgreSQL | 每日全量 + WAL 归档 PITR |
| workdirs/Skills | 每日增量 rclone |
| Vault | 每日 snapshot |

目标：RTO < 1h，RPO < 5min。

### 5.3 降级矩阵

| 故障 | 降级 |
|------|------|
| LLM Provider  down | Fallback 链 → 「服务繁忙」 |
| PostgreSQL | 只读历史对话 |
| Redis | 内存限流 |
| IntentGate 断连 | Web UI 仍可用 |

## 7. 安全合规（概要）

| 框架 | 要点 |
|------|------|
| GDPR | PII 加密、导出/删除权 |
| SOC 2 | 审计完整性、访问审查 |
| ISO 27001 | 资产清单、风险评估 |

防护分层：网络 WAF/TLS → 应用 JWT/RBAC/RLS → 数据加密 → 容器 seccomp/AppArmor。

## 9. 与内部版关系

| 内部版（当前） | 企业版（延后） |
|----------------|----------------|
| API Key | OAuth2 + RBAC |
| 单租户 | RLS 多租户 |
| docker-compose 单机 | K8s + Helm |
| .env 密钥 | Vault |
| 手动部署 | CI/CD Canary |
| IntentGate 同机 | IntentGate 独立 HA |

**原则**：内部版 API 与数据模型设计时**预留** `tenant_id`、审计字段扩展点，避免企业化时大改。
